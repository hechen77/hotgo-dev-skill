# HotGo Dev Skill 改进设计

> 对 `hotgo-dev` skill 的 SKILL.md 进行三项重大改进：重设计各端速查、强化回答规范、新增规范检查修复功能。

## 目标

1. **各端速查**：从单文档入口表 → 多维度速查 + 场景化导航
2. **回答规范**：加硬性四步链路（读文档→自检→输出→审查），分端回答策略
3. **规范检查/修复**：诊断报告 → 用户确认 → 批量修复，C 级深度（结构+代码模式+架构），五端全覆盖

## 架构决策：按能力域重组（方案 C）

SKILL.md 按 AI 行为模式重新组织为三个 Part：

```

```

### 回答四步链路（新增 — 核心改动）

将"回答前先读文档"从建议变为**强制四步**，不可跳过任何一步：

| 步骤 | 动作 | 说明 |
|-----|------|------|
| ① 读文档 | 根据各端速查定位并读取对应 references 文档 | 必须实际读取文件，不可凭记忆。该读几个读几个 |
| ② 自检 | 对照禁止事项矩阵逐条检查方案/代码 | 如果踩中禁止清单，必须先警告后给出正确方案 |

| ④ 审查 | 写完代码后再次对照禁止矩阵逐条检查 | 发现不合规立即修正，修正后再输出最终版本 |

## Part 3：执行流程

### 新建 CRUD 模块

从现有章节移入，内容不变：三步流程（列清单 → 逐文件 → 菜单提醒）。

### 规范检查与修复（全新章节）

#### 触发方式

| 触发 | 条件 | 行为 |
|------|------|------|
| 主动触发 | 用户说"检查后端规范"/"按规范修复"/"scan and fix" | 立即进入检查流程 |
| 自动嗅探 | 用户贴代码 review 或描述操作方案 | ④审查时发现违规，主动告知并建议全量检查 |

#### 检查范围

| 范围 | 命令示例 | 内容 |
|------|---------|------|
| 全项目 | "检查整个项目的规范" | 五端全部扫描 |
| 按端 | "检查后端代码规范" | 只扫描 server/ |
| 按模块 | "检查 article 模块规范" | 该模块的后端 12 + Web 前端 4 |
| 按文件 | "检查这个文件是否符合规范" | 单文件逐条对照 |

#### 检查深度（三级，默认 Level 3）

- **Level 1 — 结构层**：文件位置、命名规范、手改生成代码、import 路径
- **Level 2 — 代码模式层**（含 L1）：禁止 API、缺少必要写法、字段引用、日志工具
- **Level 3 — 架构层**（含 L1+L2）：分层调用链、Service 注册、字典注册、路由注册、中间件/鉴权、插件通信

#### 标准检查流程（四步）

```
Step 1: 诊断扫描 → 生成结构化报告
  - 概览表（按端统计：文件数/违规数/严重/警告/建议）
  - 严重问题表（文件+行号+违规类型+当前写法+规范要求）
  - 警告表
  - 建议表

Step 2: 用户确认修复范围
  选项：全部修复 / 只修复严重+警告 / 逐项选择 / 跳过

Step 3: 批量修复
  - 逐文件修改，每修复一个标注 ✓
  - 修复后同级检查，确保无新问题
  - 完成汇总

Step 4: 修复后验证
  - 对修改过的文件重新做 Level 1-3 扫描
  - 确认零新增违规
```
