## 11.1 数据库建表约定

### 11. MySQL 与 PostgreSQL 类型对照

HotGo 同时支持 MySQL 和 PostgreSQL，两种数据库的建表语法有差异，以下是核心对照：

| 含义 | MySQL | PostgreSQL |
|------|-------|-----------|
| 自增主键 | `bigint(20) NOT NULL AUTO_INCREMENT` | `BIGSERIAL PRIMARY KEY`（或 `int8 NOT NULL DEFAULT nextval('seq')`） |
| 整型 | `bigint(20)`, `int(11)`, `tinyint(1)`, `smallint` | `BIGINT`, `INTEGER`, `SMALLINT` |
| 字符串 | `varchar(N)`, `text`, `longtext` | `VARCHAR(N)`, `TEXT` |
| 精确小数 | `decimal(10,2)` | `NUMERIC(10,2)` |
| JSON | `json` | `JSONB` |
| 日期时间 | `datetime` | `TIMESTAMP` |
| 布尔 | `tinyint(1)` | `SMALLINT`（HotGo 不用 PG bool，统一用 SMALLINT） |
| 表注释 | 行内 `COMMENT '...'` | 单独 `COMMENT ON TABLE/COLUMN ... IS '...'` |
| 引擎 | `ENGINE=InnoDB DEFAULT CHARSET=utf8mb4` | 不需要 |
| 表名引用 | 反引号 `` `table` `` | 双引号 `"table"`（或不用引号） |

### 11. 标准字段

每个业务表都必须包含以下字段。**`created_at` / `updated_at` / `deleted_at` / `created_by` / `updated_by` / `deleted_by` 由框架 ORM Hook 自动维护，业务代码中禁止手动赋值**。

| 字段 | MySQL | PostgreSQL | 必须 | 说明 |
|------|-------|-----------|------|------|
| `id` | `bigint(20) NOT NULL AUTO_INCREMENT` | `BIGSERIAL PRIMARY KEY` | ✅ | 主键 |
| `created_at` | `datetime` | `TIMESTAMP` | ✅ | 创建时间（自动写入） |
| `updated_at` | `datetime` | `TIMESTAMP` | ✅ | 更新时间（自动维护） |
| `deleted_at` | `datetime` | `TIMESTAMP` | ✅ | 软删除时间（框架自动过滤 `IS NULL`） |
| `created_by` | `bigint(20) DEFAULT 0` | `BIGINT DEFAULT 0` | 推荐 | 创建者ID（自动写入） |
| `updated_by` | `bigint(20) DEFAULT 0` | `BIGINT DEFAULT 0` | 推荐 | 更新者ID（自动维护） |
| `deleted_by` | `bigint(20) DEFAULT 0` | `BIGINT DEFAULT 0` | 推荐 | 删除者ID（自动写入） |
| `status` | `tinyint(1) DEFAULT 1` | `SMALLINT DEFAULT 1` | 推荐 | 状态：1启用 2禁用，使用 `sys_normal_disable` 字典 |
| `sort` | `int(11) DEFAULT 0` | `INTEGER DEFAULT 0` | 推荐 | 排序，值越小越靠前 |

**不含以上标准字段的普通 CRUD 表示例**：

MySQL：

```sql
CREATE TABLE `hg_test_category` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT COMMENT '分类ID',
  `name` varchar(255) NOT NULL COMMENT '分类名称',
  `short_name` varchar(128) DEFAULT NULL COMMENT '简称',
  `description` varchar(255) DEFAULT NULL COMMENT '描述',
  `sort` int(11) NOT NULL COMMENT '排序',
  `remark` varchar(255) DEFAULT NULL COMMENT '备注',
  `status` tinyint(1) DEFAULT '1' COMMENT '状态',
  `created_at` datetime DEFAULT NULL COMMENT '创建时间',
  `updated_at` datetime DEFAULT NULL COMMENT '修改时间',
  `deleted_at` datetime DEFAULT NULL COMMENT '删除时间',
  PRIMARY KEY (`id`) USING BTREE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='测试分类';
```

PostgreSQL：

```sql
CREATE TABLE hg_test_category (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    short_name VARCHAR(128),
    description VARCHAR(255),
    sort INTEGER NOT NULL,
    remark VARCHAR(255),
    status SMALLINT DEFAULT 1,
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    deleted_at TIMESTAMP
);

COMMENT ON TABLE hg_test_category IS '测试分类';
COMMENT ON COLUMN hg_test_category.id IS '分类ID';
COMMENT ON COLUMN hg_test_category.name IS '分类名称';
COMMENT ON COLUMN hg_test_category.status IS '状态';
```

### 11. 树状表规范

树状表在普通表的基础上，额外增加 `pid`、`level`、`tree` 三个字段，由框架自动维护。

**必要字段**：

| 字段 | MySQL | PostgreSQL | 说明 |
|------|-------|-----------|------|
| `pid` | `bigint(20) DEFAULT 0` | `BIGINT DEFAULT 0` | 上级ID，顶级为 0 |
| `level` | `int(11) NOT NULL DEFAULT 1` | `INTEGER NOT NULL DEFAULT 1` | 树等级，从 1 开始（顶级=1） |
| `tree` | `varchar(512)` | `VARCHAR(512)` | 关系树路径，格式 `tr_<pid1> tr_<pid2> `（⚠️ 每段后面有空格） |

**完整树状表 DDL**：

MySQL：

```sql
CREATE TABLE `hg_test_dept` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT COMMENT '部门ID',
  `pid` bigint(20) DEFAULT 0 COMMENT '上级ID',
  `level` int(11) NOT NULL DEFAULT 1 COMMENT '关系树等级',
  `tree` varchar(512) DEFAULT NULL COMMENT '关系树',
  `name` varchar(64) NOT NULL COMMENT '部门名称',
  `sort` int(11) DEFAULT 0 COMMENT '排序',
  `status` tinyint(1) DEFAULT 1 COMMENT '状态',
  `created_at` datetime DEFAULT NULL COMMENT '创建时间',
  `updated_at` datetime DEFAULT NULL COMMENT '更新时间',
  `deleted_at` datetime DEFAULT NULL COMMENT '删除时间',
  PRIMARY KEY (`id`) USING BTREE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='测试部门';
```

PostgreSQL：

```sql
CREATE TABLE hg_test_dept (
    id BIGSERIAL PRIMARY KEY,
    pid BIGINT DEFAULT 0,
    level INTEGER NOT NULL DEFAULT 1,
    tree VARCHAR(512),
    name VARCHAR(64) NOT NULL,
    sort INTEGER DEFAULT 0,
    status SMALLINT DEFAULT 1,
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    deleted_at TIMESTAMP
);
COMMENT ON TABLE hg_test_dept IS '测试部门';
```

**树数据插入示例**：

```sql
-- MySQL
INSERT INTO hg_test_dept (id, pid, level, tree, name, sort) VALUES
(1, 0, 1, '',          '总公司',       1),
(2, 1, 2, 'tr_1 ',     '研发部',       1),
(3, 1, 2, 'tr_1 ',     '销售部',       2),
(4, 2, 3, 'tr_1 tr_2 ','前端组',      1),
(5, 2, 3, 'tr_1 tr_2 ','后端组',      2);

-- PostgreSQL（注意  tr_ 后面有空格！）
INSERT INTO hg_test_dept (id, pid, level, tree, name, sort) VALUES
(1, 0, 1, '',          '总公司',       1),
(2, 1, 2, 'tr_1 ',     '研发部',       1),
(3, 1, 2, 'tr_1 ',     '销售部',       2),
(4, 2, 3, 'tr_1 tr_2 ','前端组',      1),
(5, 2, 3, 'tr_1 tr_2 ','后端组',      2);
```

**关键规则**：
- `pid` / `level` / `tree` **由系统自动维护**，禁止在编辑表单中手动设置
- 新增时只需传 `pid`，`level` 和 `tree` 由 `hgorm.AutoUpdateTree()` 自动计算
- `tree` 格式为 `tr_<ancestor_id> `，尾部有空格作为分隔符，用于快速查询某节点的所有后代：`WHERE tree LIKE 'tr_2 %'`

### 11. 菜单权限表 (`hg_admin_menu`)

这是 HotGo 中最核心也是结构最复杂的表之一，管理后台所有的菜单、页面和按钮权限。

**完整 DDL**：

```sql
-- MySQL
CREATE TABLE `hg_admin_menu` (
  `id` bigint(20) NOT NULL AUTO_INCREMENT COMMENT '菜单ID',
  `pid` bigint(20) DEFAULT 0 COMMENT '父菜单ID',
  `level` int(11) NOT NULL DEFAULT 1 COMMENT '关系树等级',
  `tree` varchar(512) DEFAULT NULL COMMENT '关系树',
  `title` varchar(64) NOT NULL COMMENT '菜单名称',
  `name` varchar(128) NOT NULL COMMENT '名称编码（唯一，用于路由匹配）',
  `path` varchar(200) DEFAULT NULL COMMENT '路由地址',
  `icon` varchar(128) DEFAULT NULL COMMENT '菜单图标',
  `type` tinyint(1) NOT NULL DEFAULT 1 COMMENT '菜单类型（1目录 2页面 3按钮）',
  `redirect` varchar(255) DEFAULT NULL COMMENT '重定向地址',
  `permissions` varchar(512) DEFAULT NULL COMMENT '菜单包含权限集合（API路由，逗号分隔）',
  `permission_name` varchar(64) DEFAULT NULL COMMENT '权限名称',
  `component` varchar(255) DEFAULT NULL COMMENT '组件路径（Vue组件路径）',
  `always_show` tinyint(1) DEFAULT 0,
  `active_menu` varchar(255) DEFAULT NULL,
  `is_root` tinyint(1) DEFAULT 0,
  `is_frame` tinyint(1) DEFAULT 1,
  `frame_src` varchar(512) DEFAULT NULL,
  `keep_alive` tinyint(1) DEFAULT 0 COMMENT '缓存该路由',
  `hidden` tinyint(1) DEFAULT 0 COMMENT '是否隐藏',
  `affix` tinyint(1) DEFAULT 0 COMMENT '是否固定',
  `sort` int(11) DEFAULT 0 COMMENT '排序',
  `remark` varchar(255) DEFAULT NULL COMMENT '备注',
  `status` tinyint(1) DEFAULT 1 COMMENT '菜单状态',
  `created_at` datetime DEFAULT NULL COMMENT '创建时间',
  `updated_at` datetime DEFAULT NULL COMMENT '更新时间',
  PRIMARY KEY (`id`) USING BTREE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='管理员_菜单权限';
```

```sql
-- PostgreSQL
CREATE TABLE hg_admin_menu (
    id BIGSERIAL PRIMARY KEY,
    pid BIGINT DEFAULT 0,
    level INTEGER NOT NULL DEFAULT 1,
    tree VARCHAR(512),
    title VARCHAR(64) NOT NULL,

    path VARCHAR(200),
    icon VARCHAR(128),
    type SMALLINT NOT NULL DEFAULT 1,
    redirect VARCHAR(255),
    permissions VARCHAR(512),
    permission_name VARCHAR(64),
    component VARCHAR(255),
    always_show SMALLINT DEFAULT 0,
    active_menu VARCHAR(255),
    is_root SMALLINT DEFAULT 0,
    is_frame SMALLINT DEFAULT 1,
    frame_src VARCHAR(512),
    keep_alive SMALLINT DEFAULT 0,
    hidden SMALLINT DEFAULT 0,
    affix SMALLINT DEFAULT 0,
    sort INTEGER DEFAULT 0,
    remark VARCHAR(255),
    status SMALLINT DEFAULT 1,
    created_at TIMESTAMP,
    updated_at TIMESTAMP
);
COMMENT ON TABLE hg_admin_menu IS '管理员_菜单权限';
```

### 11. 菜单的三种类型 (`type`)

| type | 含义 | `component` | `path` | `permissions` | 说明 |
|------|------|-----------|--------|-------------|------|
| **1** | 目录 | `LAYOUT`（顶级）或 `ParentLayout`（二级分组） | 有 | 通常为空 | 目录本身不绑定 API 权限，只是菜单分组容器 |
| **2** | 页面 | Vue 组件路径如 `/addons/xxx/index` | 有 | 有，逗号分隔 | 实际可访问的页面，绑定了 API 列表权限 |
| **3** | 按钮 | 固定为 `/` | 可为空 | **核心**：写入可操作的 API 路由 | 控制页面上的编辑/删除/导出等按钮是否可见 |

### 11. 菜单层级与按钮权限绑定

一个完整的业务模块菜单树包含以下层级：

```
第1层 (type=1, level=1, pid=0)
  └── 第2层 (type=1 分组 或 type=2 页面, level=2)
        └── 第3层 (type=2 页面, level=3)
              ├── 按钮1 (type=3, level=4)  permissions="/module/edit"
              ├── 按钮2 (type=3, level=4)  permissions="/module/delete"
              ├── 按钮3 (type=3, level=4)  permissions="/module/status"
              └── 按钮4 (type=3, level=4)  permissions="/module/export"
```

**完整菜单数据插入示例**（MySQL）：

```sql
-- ═══════ 第1层：顶级目录 ═══════

    redirect, permissions, permission_name, component, sort, status, created_at, updated_at)
VALUES
(3001, 0, 1, '', '内容管理', 'ContentManage', '/content', 'FolderOutlined', 1,
 '/content/article/index', '', '', 'LAYOUT', 10, 1, NOW(), NOW());

-- ═══════ 第2层：菜单分组（放在目录下） ═══════
INSERT INTO hg_admin_menu (id, pid, level, tree, title, name, path, icon, type,
    redirect, permissions, permission_name, component, sort, status, created_at, updated_at)
VALUES
(3002, 3001, 2, 'tr_3001 ', '文章管理', 'ArticleManage', '/article', '', 1,
 '/content/article/index', '', '', 'ParentLayout', 10, 1, NOW(), NOW());

-- ═══════ 第3层：列表页面 ═══════
INSERT INTO hg_admin_menu (id, pid, level, tree, title, name, path, icon, type,
    redirect, permissions, permission_name, component, sort, status, created_at, updated_at)
VALUES
(3003, 3002, 3, 'tr_3001 tr_3002 ', '文章列表', 'ArticleIndex', 'index', '', 2,
 '', '/content/article/list,/content/article/view', '', '/content/article/index',
 10, 1, NOW(), NOW());

-- ═══════ 第4层：按钮权限（挂在列表页面下） ═══════

    permissions, component, sort, status, created_at, updated_at) VALUES
(3004, 3003, 4, 'tr_3001 tr_3002 tr_3003 ', '编辑文章', 'ArticleEdit',
 '', '', 3, '/content/article/edit', '/', 10, 1, NOW(), NOW()),

(3005, 3003, 4, 'tr_3001 tr_3002 tr_3003 ', '删除文章', 'ArticleDelete',
 '', '', 3, '/content/article/delete', '/', 20, 1, NOW(), NOW()),

(3006, 3003, 4, 'tr_3001 tr_3002 tr_3003 ', '状态切换', 'ArticleStatus',

(3007, 3003, 4, 'tr_3001 tr_3002 tr_3003 ', '导出文章', 'ArticleExport',
 '', '', 3, '/content/article/export', '/', 40, 1, NOW(), NOW());
```

**按钮权限工作原理**：

1. 后台管理员在"权限管理 → 角色权限"中为某个角色勾选菜单（包含按钮）
2. 系统将按钮的 `permissions` 值（如 `/content/article/edit`）写入 Casbin 规则表 `hg_admin_role_casbin`
3. 用户访问时，`AdminAuth` 中间件调用 `AdminRole.Verify(ctx, path, method)` 查询 Casbin
4. 前端用 `hasPermission(['/content/article/edit'])` 控制按钮是否显示

**和 Casbin 的关系**：

```sql
-- hg_admin_role_casbin（Casbin 标准 RBAC 模型）
-- p_type = 'p' 表示 policy，v0=角色标识，v1=API路径，v2=请求方法
INSERT INTO hg_admin_role_casbin (p_type, v0, v1, v2) VALUES
('p', 'super_admin', '/content/article/list',   'GET'),
('p', 'super_admin', '/content/article/view',   'GET'),
('p', 'super_admin', '/content/article/edit',   'POST'),
('p', 'super_admin', '/content/article/delete', 'POST'),
('p', 'super_admin', '/content/article/status', 'POST'),
('p', 'super_admin', '/content/article/export', 'GET');
```

这些规则由系统在"角色分配菜单"时自动写入，**不需要手动写 Casbin SQL**。

### 11. 角色-菜单绑定 (`hg_admin_role_menu`)

```sql
-- MySQL
CREATE TABLE `hg_admin_role_menu` (
  `role_id` bigint(20) NOT NULL COMMENT '角色ID',
  `menu_id` bigint(20) NOT NULL COMMENT '菜单ID',
  PRIMARY KEY (`role_id`,`menu_id`) USING BTREE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='角色与菜单关联';
```

**绑定示例**：

```sql
-- 超级管理员（role_id=1）拥有所有菜单
INSERT INTO hg_admin_role_menu (role_id, menu_id) VALUES
(1, 2047), (1, 2048), (1, 3001), (1, 3002), (1, 3003), (1, 3004);

-- 普通编辑角色（role_id=3）只拥有内容管理下的文章相关权限
INSERT INTO hg_admin_role_menu (role_id, menu_id) VALUES
(3, 3001), (3, 3002), (3, 3003),       -- 目录 + 页面
(3, 3004), (3, 3007);                    -- 只给编辑和导出，不给删除和状态
```

### 11.0 特殊字段默认行为（代码生成器自动处理）

| 数据库特征 | 表单组件 | 验证规则 |
|-----------|---------|---------|
| `status` (任意数字类型) | Select 下拉 | 使用 `sys_normal_disable` 字典 |
