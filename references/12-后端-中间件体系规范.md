## 12.1 中间件体系（Middleware）

### 中间件架构总览

HotGo 的中间件基于 GoFrame 的 `ghttp.HandlerFunc` 模式实现，所有中间件方法定义在 `sMiddleware` 结构体上，通过 `service.RegisterMiddleware()` 注册为全局服务。

**源码位置**：

```
server/internal/logic/middleware/
├── init.go              # sMiddleware 结构体 + 注册 + Ctx/CORS/DemoLimit/Addon/DeliverUserContext + IsExceptAuth/IsExceptLogin
├── admin_auth.go        # 后台鉴权中间件
├── api_auth.go          # API 鉴权中间件
├── home_auth.go         # 前台页面鉴权中间件
├── weboscket_auth.go    # WebSocket 鉴权中间件
├── pre_filter.go        # 请求输入预处理
├── response.go          # 响应后处理（JSON/XML/HTML 格式化 + 错误过滤）
├── limit_blacklist.go   # IP 黑名单限制
└── limit_develop.go     # 开发工具 IP 白名单过滤
```

### sMiddleware 结构体

```go
type sMiddleware struct {
    LoginUrl         string  // 登录路由地址
    DemoWhiteList    g.Map   // 演示模式放行的路由白名单
    NotRecordRequest g.Map   // 不记录请求体的路由（上传等大数据请求）
}

func init() {
    service.RegisterMiddleware(NewMiddleware())
}

func NewMiddleware() *sMiddleware {
    return &sMiddleware{
        LoginUrl: "/common",
        DemoWhiteList: g.Map{
            "/admin/site/accountLogin": struct{}{},  // 登录接口必须放行
            "/admin/site/mobileLogin":  struct{}{},
        },
        NotRecordRequest: g.Map{
            "/admin/upload/file":       struct{}{},  // 上传不记录 body
            "/admin/upload/uploadPart": struct{}{},
        },
    }
}
```

### 中间件执行顺序

在 `server/internal/cmd/http.go` 中绑定的全局中间件链：

```go
// ⚠️ 顺序不可随意调整，存在依赖关系！
s.BindMiddleware("/*any", []ghttp.HandlerFunc{
    service.Middleware().Ctx,             // ① 必须第一个：初始化请求上下文（i18n、链路追踪）
    service.Middleware().CORS,            // ② 跨域处理
    service.Middleware().Blacklist,       // ③ IP 黑名单拦截
    service.Middleware().DemoLimit,       // ④ 演示模式 POST 限制
    service.Middleware().PreFilter,       // ⑤ 请求输入预处理（Filter 接口）
    service.Middleware().ResponseHandler, // ⑥ 响应后处理（在所有业务逻辑之后执行）
}...)
```

路由组级别的鉴权中间件在 `router/admin.go` 等文件中按需绑定：

```go
group.Group("/admin", func(group *ghttp.RouterGroup) {
    // 不需要登录的接口
    group.Bind(common.Site)                 // 登录接口

    // ⭐ 需要登录 + 权限验证
    group.Middleware(service.Middleware().AdminAuth)
    group.Bind(
        sys.Config, admin.Member, // ... 受保护的控制器
    )

    // ⭐ 需要登录 + 开发工具白名单
    group.Middleware(service.Middleware().Develop)
    group.Bind(sys.GenCodes, sys.Addons)
})
```

### 中间件分类

| 类别 | 中间件 | 执行位置 | 作用 |
|------|--------|---------|------|
| **全局** | `Ctx` | 所有请求最前 | 初始化上下文：i18n 语言、链路追踪 Jaeger、contexts.Init |
| **全局** | `CORS` | 所有请求 | 自动处理跨域请求头 |
| **全局** | `Blacklist` | 所有请求 | 校验请求 IP 是否在黑名单中 |
| **全局** | `DemoLimit` | 所有请求 | 演示模式下拒绝非白名单 POST 请求 |
| **全局** | `PreFilter` | 所有请求 | 自动调用实现了 `validate.Filter` 接口的 Req 的 `Filter()` 方法 |
| **全局** | `ResponseHandler` | 所有请求之后 | 统一响应格式化（JSON/XML/HTML）+ 错误信息过滤 |
| **路由组** | `AdminAuth` | `/admin` 路由组 | JWT 解析 + Casbin RBAC 权限验证 |
| **路由组** | `ApiAuth` | `/api` 路由组 | JWT 解析（可配置豁免路径） |
| **路由组** | `HomeAuth` | `/home` 路由组 | 前台页面鉴权 |
| **路由组** | `WebSocketAuth` | `/socket` 路由组 | WebSocket 连接鉴权 |
| **路由组** | `Addon` | 插件路由组 | 从 URL 解析插件模块名写入 context |
| **路由组** | `Develop` | 开发工具路由组 | IP 白名单校验（只允许配置的 IP 访问代码生成器等） |
| **辅助** | `DeliverUserContext` | 由鉴权中间件调用 | JWT 解析 → 按 App 类型绑定用户到 context |
| **辅助** | `IsExceptAuth` | 由鉴权中间件调用 | 检查路径是否在 `config.yaml` 的 `router.xxx.exceptAuth` 豁免列表中 |
| **辅助** | `IsExceptLogin` | 由鉴权中间件调用 | 检查路径是否在 `config.yaml` 的 `router.xxx.exceptLogin` 豁免列表中 |

### 12. Ctx — 请求上下文初始化

**文件**：`init.go`

```go
func (s *sMiddleware) Ctx(r *ghttp.Request) {
    // 1. 国际化：从 Header 解析语言（zh-CN / zh-TW / en）
    r.SetCtx(gi18n.WithLanguage(r.Context(), simple.GetHeaderLocale(r.Context())))

    // 2. 链路追踪：如果开启 Jaeger，创建 Span
    if g.Cfg().MustGet(r.Context(), "jaeger.switch").Bool() {
        ctx, span := gtrace.NewSpan(r.Context(), "middleware.ctx")
        span.SetAttributes(attribute.KeyValue{Key: "traceID", Value: ...})
        span.End()
        r.SetCtx(ctx)
    }

    // 3. 初始化上下文数据：请求体、模块名
    contexts.Init(r, &model.Context{
        Data:   data,
        Module: getModule(r.URL.Path),  // 从 URL 第一段提取模块名：/admin/xxx → admin
    })

    // 4. 设置 SessionId（用于日志追踪）
    if len(r.Cookie.GetSessionId()) == 0 {
        r.Cookie.SetSessionId(gctx.CtxId(r.Context()))
    }

    // 5. 设置永不超时的 context（GoFrame 特性）
    r.SetCtx(r.GetNeverDoneCtx())

    r.Middleware.Next()
}
```

**关键设计**：
- 必须是**第一个**全局中间件，后续所有中间件依赖它初始化的 context
- 国际化语言从 HTTP Header `Accept-Language` 中自动解析
- `NotRecordRequest` 白名单中的路由（如文件上传）不记录请求体，避免大请求体影响性能
- `r.GetNeverDoneCtx()` 确保即使客户端断开连接，服务端也能完成日志写入等善后工作

### 12. PreFilter — 请求输入预处理

**文件**：`pre_filter.go`

```go
func (s *sMiddleware) PreFilter(r *ghttp.Request) {
    router := global.GetRequestRoute(r)
    if router == nil { r.Middleware.Next(); return }

    funcInfo := router.Handler.Info
    // 非规范路由（不是 XxxReq/XxxRes 模式）不处理
    if funcInfo.Type.NumIn() != 2 { r.Middleware.Next(); return }

    // 反射创建 Req 实例
    inputType := funcInfo.Type.In(1)
    var inputObject reflect.Value
    if inputType.Kind() == reflect.Ptr {
        inputObject = reflect.New(inputType.Elem())
    } else {
        inputObject = reflect.New(inputType.Elem()).Elem()
    }

    // 1. 先执行 GoFrame 内置的 g.Validator 校验（v:"required" 等 tag）
    if err := r.Parse(inputObject.Interface()); err != nil {
        response.JsonExit(r, gerror.Code(err).Code(), gerror.Current(err).Error())
        return
    }

    // 2. 如果 Req 实现了 validate.Filter 接口，自动调用 Filter()
    if _, ok := inputObject.Interface().(validate.Filter); !ok {
        r.Middleware.Next()
        return
    }
    if err := validate.PreFilter(r.Context(), inputObject.Interface()); err != nil {
        response.JsonExit(r, gerror.Code(err).Code(), gerror.Current(err).Error())
        return
    }

    // 3. 将过滤后的参数写回请求
    r.SetParamMap(gconv.Map(inputObject.Interface()))
    r.Middleware.Next()
}
```

**设计亮点**：
- 使用**反射**自动检测每个 Req 是否实现了 `validate.Filter` 接口
- 开发者无需手动在 Controller 中调用 `Filter()`，完全透明
- 先执行框架内置校验 → 再执行业务 Filter → 将处理后的参数写回

### 12. DeliverUserContext — 用户信息绑定核心机制

**文件**：`init.go`

这是整个鉴权体系的**核心方法**，被 `AdminAuth`、`ApiAuth`、`HomeAuth`、`WebSocketAuth` 四个鉴权中间件共同调用。

```go
func (s *sMiddleware) DeliverUserContext(r *ghttp.Request) (err error) {
    // ⭐ Step 1: 从 HTTP 请求中解析 JWT Token，得到 model.Identity
    user, err := token.ParseLoginUser(r)
    if err != nil {
        return  // Token 无效或过期 → 返回错误，由调用方 response.JsonExit
    }

    // ⭐ Step 2: 根据 JWT 中携带的 App 字段，走不同的用户绑定策略
    switch user.App {
    case consts.AppAdmin:
        // Admin 端：从 admin_member 表加载用户 → 校验状态 → 查角色/部门 → 拼完整 Identity → SetUser
        if err = service.AdminSite().BindUserContext(r.Context(), user); err != nil {
            return
        }
    default:
        // Api / Home / WebSocket 端：JWT 中的 Identity 已包含登录时写入的全部字段，直接绑定
        contexts.SetUser(r.Context(), user)
    }
    return
}
```

**核心设计**：`token.ParseLoginUser(r)` 解析出的 `model.Identity` 里有一个关键字段 `App`，它在登录时被写入。

**HotGo 原始版 Identity**（`server/internal/model/context.go`）：

```go
// Identity 通用身份模型
type Identity struct {
    Id       int64       `json:"id"              description:"用户ID"`
    Pid      int64       `json:"pid"             description:"上级ID"`
    DeptId   int64       `json:"deptId"          description:"部门ID"`
    DeptType string      `json:"deptType"        description:"部门类型"`
    RoleId   int64       `json:"roleId"          description:"角色ID"`
    RoleKey  string      `json:"roleKey"         description:"角色唯一标识符"`
    Username string      `json:"username"        description:"用户名"`
    RealName string      `json:"realName"        description:"姓名"`
    Avatar   string      `json:"avatar"          description:"头像"`
    Email    string      `json:"email"           description:"邮箱"`
    Mobile   string      `json:"mobile"          description:"手机号码"`
    App      string      `json:"app"             description:"登录应用"`
    LoginAt  *gtime.Time `json:"loginAt"         description:"登录时间"`
}
```

**⚠️ Identity 可按业务自定义扩展**。`model.Identity` 是 HotGo 预留的**可扩展结构体**——原始版只包含后台管理系统（Admin）通用的字段。如果你的项目有 Home 端（移动 App）、Api 端（外部脚本）等不同 App 类型，可以在 Identity 上按需追加业务专属字段。

以下是实际项目中的扩展示例（新增了 6 个字段）：

```go
// Identity 通用身份模型（根据业务扩展后）
type Identity struct {
    // ========== HotGo 原始字段 ==========
    Id       int64       `json:"id"`
    Pid      int64       `json:"pid"`             // 上级ID（Admin 用）
    DeptId   int64       `json:"deptId"`           // 部门ID（Admin 用）
    DeptType string      `json:"deptType"`         // 部门类型：company/tenant/merchant/user
    RoleId   int64       `json:"roleId"`           // 角色ID（Admin 用）
    RoleKey  string      `json:"roleKey"`          // 角色唯一编码（Casbin RBAC 需要）
    Username string      `json:"username"`
    RealName string      `json:"realName"`
    Avatar   string      `json:"avatar"`
    Email    string      `json:"email"`
    Mobile   string      `json:"mobile"`
    App      string      `json:"app"`              // ⭐ 来源应用：admin / api / home / websocket
    LoginAt  *gtime.Time `json:"loginAt"`

    // ========== 业务扩展字段（示例） ==========
    LevelId     int64    `json:"levelId"`          // 业务身份ID（Home 端需要）
    PackageId   int64    `json:"packageId"`        // 业务套餐ID（Home 端需要，按项目扩展）
    Platform    string   `json:"platform"`         // 登录平台（mobile/desktop/mini_program）
    Version     string   `json:"version"`          // App 版本号
    Permissions []string `json:"permissions"`      // 功能权限列表（Home 端业务权限，非 Casbin RBAC）
    IsScript    bool     `json:"isScript"`         // 是否为脚本登录（Api 端用）
}
```

**扩展原则**：
- **只增不减**：不要删除 HotGo 原始字段（`Id`、`App`、`RoleKey` 等被框架内部多处引用），只能追加新字段
- **按 App 类型使用**：Admin 端的登录逻辑只填充 Admin 相关字段；Home 端在登录时额外填充 `LevelId`、`PackageId`、`Permissions`
- **JWT 自动携带**：`token.Login(ctx, user)` 会将整个 Identity 序列化进 JWT Payload，扩展的字段自动跟随 Token 传递，`token.ParseLoginUser(r)` 解析时自动还原

**两种绑定策略的差异**：

| 对比维度 | `AppAdmin` → `BindUserContext` | `AppApi` / `AppHome` / `AppWebSocket` → `contexts.SetUser` |
|---------|-------------------------------|-----------------------------------------------------------|
| **数据来源** | 每次请求实时从 `admin_member` 表查 | JWT Token 中自带（登录时写入，不查库） |
| **角色信息** | ✅ 查 `admin_role` 表获取 RoleKey | ❌ 无角色概念，JWT 中不需要 |
| **部门信息** | ✅ 查 `admin_dept` 表获取 DeptType | ❌ 无部门概念 |
| **状态校验** | ✅ 校验用户/角色/部门是否启用 | ❌ 由登录端自己的业务逻辑管理 |
| **权限校验** | ✅ `AdminRole.Verify()` Casbin RBAC | ❌ 不校验路由权限 |
| **性能** | 每次请求多 3 次 DB 查询 | 纯 JWT 解析，无额外 DB 查询 |

**为什么要这样区分？**

- **Admin 端**是后台管理系统，权限敏感。每次请求实时查库确保：角色被禁用后立即失效、权限变更后实时生效。代价是 3 次额外 DB 查询。
- **Api 端**是脚本设备/桌面端，高频调用。只校验登录态，不做权限检查。JWT 中的身份信息在登录时已验证并写入。
- **Home 端**是移动 App 用户，使用的是业务权限体系（不是 Casbin RBAC），Login 时已将 `Permissions`、`LevelId`、`PackageId` 写入 Identity。

**Admin 端的 `BindUserContext` 实现**：

`server/internal/logic/admin/site.go`：

```go
func (s *sAdminSite) BindUserContext(ctx context.Context, claims *model.Identity) (err error) {
    // 1. 从数据库加载用户，校验账号是否存在 + 是否启用
    var mb *entity.AdminMember
    if err = dao.AdminMember.Ctx(ctx).WherePri(claims.Id).Scan(&mb); err != nil {
        return gerror.Wrap(err, "获取用户信息失败")
    }
    if mb == nil { return gerror.New("账号不存在或已被删除") }
    if mb.Status != consts.StatusEnabled { return gerror.New("账号已被禁用") }

    // 2. 加载角色信息，校验角色是否存在 + 是否启用 → 获取 RoleKey
    role, dept, err := s.getLoginRoleAndDept(ctx, mb.RoleId, mb.DeptId)
    if err != nil { return err }

    // 3. 组装完整的 Identity（从 DB 获取最新数据，覆盖 JWT 中的旧值）
    user := &model.Identity{
        Id:       mb.Id,
        Pid:      mb.Pid,
        DeptId:   dept.Id,
        DeptType: dept.Type,     // ⭐ 用于数据权限过滤（company/tenant/merchant/user）
        RoleId:   mb.RoleId,
        RoleKey:  role.Key,      // ⭐ 用于 Casbin RBAC 验证
        Username: mb.Username,
        RealName: mb.RealName,
        Avatar:   mb.Avatar,
        Email:    mb.Email,
        Mobile:   mb.Mobile,
        App:      claims.App,
        LoginAt:  claims.LoginAt,
    }

    // 4. 写入 context（后续 logic 层通过 contexts.GetUser(ctx) 获取）
    contexts.SetUser(ctx, user)
    return
}
```

**Home 端 / Api 端登录时构建 Identity 的通用示例**：

以下是一个完整的 Home 端登录逻辑示例，展示如何根据业务需求构建 Identity 并通过 `token.Login` 生成 JWT：

```go
// addons/<your_addon>/logic/<app_type>/user.go
func (s *sUserService) UserLogin(ctx context.Context, in *LoginInp) (token string, expires int64, err error) {
    // 1. 从用户表查询
    var userInfo *entity.User
    if err = dao.User.Ctx(ctx).Where("username", in.Username).Scan(&userInfo); err != nil {
        return
    }
    if userInfo == nil {
        err = gerror.New("用户名或密码错误")
        return
    }

    // 2. 验证密码
    if err = simple.CheckPassword(in.Password, userInfo.Salt, userInfo.PasswordHash); err != nil {
        return
    }

    // 3. ⭐ 构建 Identity（填充 HotGo 原始字段 + 业务扩展字段）
    identity := &model.Identity{
        // HotGo 原始字段
        Id:       userInfo.Id,
        Username: userInfo.Username,
        RealName: userInfo.RealName,
        Mobile:   userInfo.Mobile,
        Avatar:   userInfo.Avatar,
        App:      consts.AppHome,   // ⭐ 决定 DeliverUserContext 走哪个分支
        LoginAt:  gtime.Now(),

        // 业务扩展字段（根据你的 model.Identity 定义来填）
        Platform:    in.Platform,            // mobile / desktop / mini_program
        Version:     in.Version,             // App 版本号
        LevelId:     memberInfo.LevelId,      // 业务身份
        PackageId:   memberInfo.PackageId,    // 业务套餐
        Permissions: loadUserPermissions(ctx, userInfo.Id), // 业务功能权限
    }

    // 4. ⭐ token.Login 将整个 Identity 序列化进 JWT
    //    扩展字段（LevelId、Permissions 等）会自动带入 JWT Payload
    token, expires, err = token.Login(ctx, identity)
    return
}
```

**Api 端（脚本/桌面端）登录的构建方式类似，只需将 `App` 设为 `consts.AppApi`**：

```go
identity := &model.Identity{
    Id:       deviceInfo.Id,
    Username: deviceInfo.Name,
    App:      consts.AppApi,       // ⭐ Api 端
    LoginAt:  gtime.Now(),
    IsScript: true,                 // 业务扩展字段
}
token, expires, err = token.Login(ctx, identity)
```

**关键要点**：
- `App` 字段决定 `DeliverUserContext` 走查库绑定（Admin）还是直接绑定（其他）
- `token.Login()` 会将整个 Identity（含扩展字段）序列化为 JWT
- `token.ParseLoginUser()` 解析时自动还原所有字段
- 增删 Identity 字段后，旧的 JWT Token 会解析失败（字段不匹配），用户需重新登录

### 12. AdminAuth — 后台鉴权（JWT + Casbin RBAC）

**文件**：`admin_auth.go`

```go
func (s *sMiddleware) AdminAuth(r *ghttp.Request) {
    ctx := r.Context()
    // ⭐ 去掉路由前缀 /admin → 拿到纯业务路径
    path := gstr.Replace(r.URL.Path, simple.RouterPrefix(ctx, consts.AppAdmin), "", 1)

    // Step 1：检查是否在「免登录」名单中（如登录接口本身）
    if s.IsExceptLogin(ctx, consts.AppAdmin, path) {
        r.Middleware.Next()
        return
    }

    // Step 2：JWT 解析 → 走 Admin 专属的 BindUserContext（查库+验角色+验部门）
    if err := s.DeliverUserContext(r); err != nil {
        response.JsonExit(r, gcode.CodeNotAuthorized.Code(), err.Error())
        return
    }

    // Step 3：检查是否在「免权限验证」名单中（如首页、个人信息）
    if s.IsExceptAuth(ctx, consts.AppAdmin, path) {
        r.Middleware.Next()
        return
    }

    // Step 4：⭐ Casbin RBAC 权限验证（Admin 独有！）
    if !service.AdminRole().Verify(ctx, path, r.Method) {
        response.JsonExit(r, gcode.CodeSecurityReason.Code(), "你没有访问权限！")
        return
    }

    r.Middleware.Next()
}
```

**鉴权流程图**：

```
请求 /admin/member/list
  → /admin 前缀剥离 → path = "/member/list"
  → IsExceptLogin(admin, "/member/list")? → 否 → 继续
  → DeliverUserContext:
      → token.ParseLoginUser(r)  → JWT 中的 App = "admin"
      → service.AdminSite().BindUserContext(ctx, user):
          → 查 admin_member 表 → 验状态
          → 查 admin_role 表 → 获取 RoleKey
          → 查 admin_dept 表 → 获取 DeptType
          → contexts.SetUser(ctx, 完整 Identity)
  → IsExceptAuth(admin, "/member/list")? → 否 → 继续
  → AdminRole.Verify(ctx, "/member/list", "GET"):
      → Casbin Enforcer.Enforce(user.RoleKey, "/member/list", "GET")
      → ✅ 有权限 → Next()
      → ❌ 无权限 → JsonExit(62, "你没有访问权限！")
```

**何时使用 AdminAuth**：仅用于 `/admin` 路由组，需要后台管理员账号 + Casbin RBAC 角色权限。

**在 router/admin.go 中的挂载方式**：


### 12. ApiAuth — API 鉴权（JWT + 免权限）

**文件**：`api_auth.go`

```go
func (s *sMiddleware) ApiAuth(r *ghttp.Request) {
    ctx := r.Context()
    path := gstr.Replace(r.URL.Path, simple.RouterPrefix(ctx, consts.AppApi), "", 1)

    // Step 1：检查是否免登录（如支付回调、脚本登录）
    if s.IsExceptLogin(ctx, consts.AppApi, path) {
        r.Middleware.Next()
        return
    }

    // Step 2：JWT 解析 → 走 default 分支 → contexts.SetUser（简单绑定）
    if err := s.DeliverUserContext(r); err != nil {
        response.JsonExit(r, gcode.CodeNotAuthorized.Code(), err.Error())
        return
    }

    // ⭐ 不调用 AdminRole.Verify() —— Api 端不做路由权限验证！
    r.Middleware.Next()
}
```

**与 AdminAuth 的关键差异**：

| 对比 | AdminAuth | ApiAuth |
|------|-----------|---------|
| 登录豁免 | `router.admin.exceptLogin` | `router.api.exceptLogin` |
| 用户绑定 | `AdminSite().BindUserContext()`（查库 + 验角色部门） | `contexts.SetUser()`（JWT 信息直接绑定） |
| 权限验证 | ✅ `AdminRole().Verify()` Casbin RBAC | ❌ 不验证路由权限 |
| 典型调用方 | 浏览器（管理员登录后台） | 数据采集端、桌面客户端、第三方系统 |
| JWT 中的 App | `AppAdmin` | `AppApi` |

**何时使用 ApiAuth**：用于 `/api` 路由组。调用方通常是外部脚本程序、桌面客户端、第三方服务，只需要登录态，不需要后台的 Casbin 角色权限。

**典型场景**：
- 数据采集端上报业务数据（`/api/data/report/add`）
- 桌面端查询业务数据（`/api/data/list`）
- 第三方系统回调（`/api/pay/notify` — 通常在 `exceptLogin` 中豁免）

**在 router/api.go 中的挂载方式**：

```go

        // 不需要登录的

        // ⭐ 需要登录（仅验 JWT，不验权限）
        group.Middleware(service.Middleware().ApiAuth)
        group.Bind(member.NewV1()) // 其他 API
    })
}
```

### 12. WebSocketAuth — WebSocket 鉴权

**文件**：`weboscket_auth.go`

```go
func (s *sMiddleware) WebSocketAuth(r *ghttp.Request) {
    ctx := r.Context()
    path := gstr.Replace(r.URL.Path, simple.RouterPrefix(ctx, consts.AppWebSocket), "", 1)

    // Step 1：检查是否免登录
    if s.IsExceptLogin(ctx, consts.AppWebSocket, path) {
        r.Middleware.Next()
        return
    }

    // Step 2：JWT 解析（Token 从 URL 参数 ?authorization=xxx 中获取）
    if err := s.DeliverUserContext(r); err != nil {
        response.JsonExit(r, gcode.CodeNotAuthorized.Code(), err.Error())
        return
    }

    // ⭐ 不做路由权限验证 —— WebSocket 的权限由消息 Router 中的 EventHandler 各自控制
    r.Middleware.Next()
}
```

**WebSocket 鉴权的特殊性**：

- WebSocket 连接时 Token 通过 **URL 查询参数** 传递（`ws://host/socket?authorization=xxxx`），而非 HTTP Header
- `token.ParseLoginUser(r)` 内部会自动从 Header 或 URL 参数中提取 Token
- 连接建立后，后续的消息权限由 `websocket.EventHandler` 在业务层各自判断
- 不调用 `AdminRole().Verify()` —— WS 是按事件路由而非按 URL 路由

**何时使用 WebSocketAuth**：用于 `/socket` 路由组。

### 12. Identity.App 填写规范

**文件**：`server/internal/consts/app.go`

```go
// 应用类型
const (
    AppAdmin     = "admin"      // 后台管理系统
    AppApi       = "api"        // 外部 API（脚本、桌面端、第三方）
    AppHome      = "home"       // 前台页面（移动端 App、H5）
    AppWebSocket = "websocket"  // WebSocket 连接
    AppDefault   = "default"    // 兜底（URL 无法识别模块时）
)
```

**`App` 值贯穿整个请求生命周期**，它同时控制三件事：

| 影响范围 | 机制 | 说明 |
|---------|------|------|
| **路由前缀** | `RouterPrefix(ctx, app)` → 读 `config.yaml` 的 `router.<app>.prefix` | `/admin/xxx`、`/api/xxx`、`/home/xxx`、`/socket/xxx` |
| **鉴权方式** | 路由组绑定 `AdminAuth` / `ApiAuth` / `HomeAuth` / `WebSocketAuth` | 决定了 JWT 解析后是否还需要 Casbin RBAC |
| **用户绑定** | `DeliverUserContext` 中的 `switch user.App` | `AppAdmin` → `BindUserContext`（查库，完整身份）；其他 → `contexts.SetUser`（JWT 直绑） |

**登录时必须严格填写 `App`**：

```go
// ✅ Admin 后台管理员登录
user := &model.Identity{
    // ...其他字段...
    App: consts.AppAdmin,   // ⚠️ 必须与路由模块匹配
}
token.Login(ctx, user)

// ✅ Home 移动端 App 用户登录
user := &model.Identity{
    // ...其他字段...
    App: consts.AppHome,
}
token.Login(ctx, user)

// ✅ Api 外部脚本/桌面端登录
user := &model.Identity{
    // ...其他字段...
    App: consts.AppApi,
}
token.Login(ctx, user)
```

**填写规则**：

| 用户来源 | 应填 `App` 值 | 登录接口路径 | 后续请求路径 | 鉴权 |
|---------|-------------|------------|------------|------|
| Web 后台管理页面 | `consts.AppAdmin` | `/admin/site/accountLogin` | `/admin/xxx` | JWT + Casbin RBAC |
| 移动端 Flutter App | `consts.AppHome` | `/home/user/login` | `/home/xxx` | JWT 仅登录 |
| 微信小程序 | `consts.AppHome` | `/home/applet/user/login` | `/home/xxx` | JWT 仅登录 |
| PC 桌面客户端 | `consts.AppApi` | `/api/device/login` | `/api/xxx` | JWT 仅登录 |
| 外部脚本/设备 | `consts.AppApi` | `/api/device/login` | `/api/xxx` | JWT 仅登录 |
| WebSocket 连接 | 任意 | 通过 URL 参数 `?authorization=token` | `/socket` | JWT 仅登录 |

**⚠️ 常见错误**：

| 错误 | 后果 |
|------|------|
| 移动端登录时 `App` 填了 `admin` | `DeliverUserContext` 走 `AdminSite().BindUserContext()` → 去 `admin_member` 表查用户 → 查不到，报"账号不存在" |
| 后台管理员登录时 `App` 填了 `home` | `DeliverUserContext` 走 `contexts.SetUser()` 直绑 → 没有 `RoleKey` → 后续 `AdminRole.Verify()` 失败，所有接口返回"没有访问权限" |
| `App` 值拼写错误（如 `"Home"` 而非 `"home"`） | `DeliverUserContext` 走 `default` 分支 → 直绑 → 但如果该路由组有 `AdminAuth`，也会因缺少 `RoleKey` 导致权限验证失败 |
| JWT Token 中的 `App` 与请求路径不匹配 | 例如 `App=home` 的 Token 请求 `/admin/member/list` → `AdminAuth` 走 `DeliverUserContext` → `BindUserContext` 去 `admin_member` 表查 → 该用户在 `admin_member` 中不存在 |

**✅ 正确做法**：**一个用户表对应一个 `App` 常量**。不要跨表混用：

```
admin_member  表 → App = consts.AppAdmin     → /admin 路由
app_user      表 → App = consts.AppHome      → /home  路由
device        表 → App = consts.AppApi       → /api   路由
```

如果同一个用户既能在后台登录又能在 App 登录（两个不同的账号体系），他会有两个独立的 Identity（Admin 用 `admin_member` 的 ID，Home 用 `app_user` 的 ID），分别登录获得两个不同 App 值的 JWT Token。

### 12. ResponseHandler — 响应后处理

**文件**：`response.go`

```go
func (s *sMiddleware) ResponseHandler(r *ghttp.Request) {
    r.Middleware.Next()  // ⭐ 先执行所有业务逻辑！

    // 错误状态码接管
    switch r.Response.Status {
    case 403: r.Response.Writeln("403 - 网站拒绝显示此网页"); return
    case 404: r.Response.Writeln("404 - 你似乎来到了没有知识存在的荒原…"); return
    }

    // 已存在响应（如图片/文件流）→ 不处理
    contentType := getContentType(r)
    if contentType != consts.HTTPContentTypeStream && r.Response.BufferLength() > 0 {
        return
    }

    // 按 Content-Type 分发
    switch contentType {
    case "text/html":       s.responseHtml(r)    // HTML 错误模板
    case "text/xml":        s.responseXml(r)      // XML 格式
    case "text/event-stream", "application/octet-stream":   // SSE/文件流 → 跳过
    default:                responseJson(r)        // JSON（默认）
    }
}
```

**JSON 响应格式化**：

```go
func responseJson(r *ghttp.Request) {
    code, message, data := parseResponse(r)
    response.RJson(r, code, message, data)
}

func parseResponse(r *ghttp.Request) (code int, message string, resp interface{}) {
    err := r.GetError()
    if err == nil {
        return gcode.CodeOK.Code(), "操作成功", r.GetHandlerResponse()
    }

    // Debug 模式 → 返回完整堆栈（开发调试用）
    if simple.Debug(ctx) {
        message = gerror.Current(err).Error()
        resp = charset.ParseErrStack(err)
    } else {
        // 生产模式 → 只返回最外层友好提示
        message = consts.ErrorMessage(gerror.Current(err))
    }

    // 记录异常日志
    code = gerror.Code(err).Code()
    if code == gcode.CodeNil.Code() {
        g.Log().Stdout(false).Infof(ctx, "exception:%v", err)  // 可控错误：仅文件日志
    } else {
        g.Log().Errorf(ctx, "exception:%v", err)                // 不可控：文件+服务日志+堆栈
    }
    return
}
```

**何时使用 ResponseHandler**：已作为全局中间件绑定在 `cmd/http.go` 中，**无需手动调用**。它是所有请求的最后一个处理环节。

**关键设计**：
- `r.Middleware.Next()` **先于**响应逻辑执行——业务处理完才走到这里
- Debug 模式返回完整堆栈，生产模式只返回友好提示
- 错误码 `-1`（`CodeNil`）仅记文件日志，非 `-1` 的错误同时记服务日志
- SSE / 文件流 / 已渲染的 HTML 响应 → 跳过 JSON 包装

```go
}
```

**Step 2**：在 `service/middleware.go` 的 `IMiddleware` 接口中添加方法声明（或运行 `gf gen service` 自动生成）：

```go
type IMiddleware interface {
    // ...existing methods...
    RateLimit(r *ghttp.Request)
}
```

**Step 3**：在对应的位置绑定：

```go
// 全局绑定（cmd/http.go → 所有路由生效）
s.BindMiddleware("/*any", []ghttp.HandlerFunc{
    // ...
    service.Middleware().RateLimit,  // 按需加在合适位置
}...)

// 路由组绑定（router/admin.go → 只有 /admin 路由生效）
group.Middleware(service.Middleware().RateLimit)
```

### 中间件编写规范速查

| 规范 | 说明 |
|------|------|
| 方法绑定在 `sMiddleware` 上 | 统一入口，便于访问配置（DemoWhiteList、NotRecordRequest 等） |
| 正常放行用 `r.Middleware.Next()` | 不调用 Next 则中断整个中间件链 |
| 拒绝请求用 `response.JsonExit()` | 统一响应格式 `{code, message, timestamp, traceID}` |
| 鉴权中间件先用 `gstr.Replace` 去路由前缀 | 拿到纯业务路径再校验 |
| 豁免路径从 `config.yaml` 读取 | 通过 `IsExceptAuth(adminApiHome..., path)` / `IsExceptLogin(..., path)` 判断，不硬编码 |
| 新增中间件后运行 `gf gen service` | 自动更新 `service/middleware.go` 中的 `IMiddleware` 接口 |
| 全局中间件按依赖顺序排列 | `Ctx` 必须第一（后续都依赖它初始化的 context）；`ResponseHandler` 必须最后（它包裹所有业务逻辑）
